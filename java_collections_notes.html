Iterable<T>

Root of the Collection tree — java.lang
interface Iterable<T> ▶

The topmost interface in the Collection hierarchy, located in java.lang. Any class implementing Iterable can be used in a for-each loop.

📦 Package: java.lang 🔄 Enables for-each loop 📋 Single abstract method
Core Method
public interface Iterable<T> {
    Iterator<T> iterator();  // returns an Iterator over elements

    // Default methods added in Java 8
    default void forEach(Consumer<? super T> action) { ... }
    default Spliterator<T> spliterator() { ... }
}
💡 Iterator pattern: iterator() returns an Iterator with hasNext(), next(), and remove(). Always use it.remove() — never collection.remove() — inside an iterator loop.

ArrayList<E>

Resizable array — most commonly used List
class ArrayList<E> ▶

Implements List using a dynamic array. Best for random access and iteration. Default choice when you need a List.

📦 Default capacity: 16 📈 Growth: 50% (×1.5) ❌ Not thread-safe ✅ Allows null 🔄 Fail-fast iterator
Time Complexity
Random Access
O(1)
Add (end)
O(1)*
Add (middle)
O(n)
Remove (end)
O(1)
Remove (middle)
O(n)
Search
O(n)

* Amortized O(1) — occasionally O(n) during resize

Common Usage
List<String> list = new ArrayList<>();        // default capacity 10
List<String> list = new ArrayList<>(50);     // set initial capacity
List<String> list = new ArrayList<>(other);   // copy constructor

list.add("Java");
list.add(0, "First");         // insert at index 0
list.get(1);                    // O(1) random access
list.set(0, "Updated");        // replace at index
list.remove(0);               // remove by index
list.remove("Java");          // remove by value (first occurrence)
list.contains("Java");        // O(n) linear search
list.sort(Comparator.naturalOrder());
list.removeIf(s -> s.isEmpty());// Java 8 — best way to remove safely

// Immutable List (Java 9+)
List<String> immutable = List.of("a", "b", "c");
⚠️ ConcurrentModificationException: Never call list.remove() inside a for-each loop. Use iterator.remove() or list.removeIf() instead.

LinkedList<E>

Doubly-linked list — implements List, Deque, Queue
class LinkedList<E> ▶

Doubly-linked list implementation of List, Deque, and Queue. Every element is a node with prev and next references. Ideal for frequent insertions/deletions at both ends.

🔗 Doubly linked nodes ❌ No random access ✅ Implements List + Deque ❌ Not thread-safe 💾 Higher memory (node overhead)
Time Complexity
Access by index
O(n)
Add/Remove (head/tail)
O(1)
Add/Remove (middle)
O(n)*
Search
O(n)

* O(n) to find node, then O(1) to unlink

Usage as List, Queue, Stack
LinkedList<Integer> ll = new LinkedList<>();

// List operations
ll.add(1);  ll.add(2);  ll.add(3);

// Deque / Queue operations
ll.addFirst(0);       // O(1)
ll.addLast(4);        // O(1)
ll.peekFirst();       // view head without removing
ll.pollFirst();       // remove and return head
ll.pollLast();        // remove and return tail

// As Stack (LIFO)
ll.push(10);          // same as addFirst
ll.pop();             // same as removeFirst

// As Queue (FIFO)
ll.offer(5);          // same as addLast
ll.poll();            // same as removeFirst

Vector & Stack

Legacy thread-safe List — avoid in new code
class Vector<E> / Stack<E> ▶

Vector is like ArrayList but fully synchronized (every method is synchronized). Stack extends Vector and adds push/pop. Both are considered legacy — avoid in modern code.

FeatureVectorArrayList
Thread-safe✅ Yes (synchronized)❌ No
Growth factorDoubles (×2)50% (×1.5)
PerformanceSlower (lock overhead)Faster
LegacyYes — Java 1.0Java 1.2 (modern)
⚠️ Prefer instead: Use CopyOnWriteArrayList or Collections.synchronizedList() for thread safety. Use ArrayDeque instead of Stack.

Set<E> Interface

No duplicates · Unordered (unless specified)
interface Set<E> ▶

Extends Collection. A collection that contains no duplicate elements. At most one null element (in HashSet). The contract relies on equals() and hashCode() for uniqueness.

❌ No duplicates ❓ Order depends on impl 🔑 equals() + hashCode() contract
ℹ️ Sub-interfaces: SortedSet<E> (adds sorting) → NavigableSet<E> (adds floor/ceiling/higher/lower ops) → implemented by TreeSet.
⚡

HashSet<E>

Hash table backed — fastest Set, no order
class HashSet<E> ▶

Backed internally by a HashMap<E, Object>. Elements are stored as keys; a dummy PRESENT object is the value. O(1) average for add/remove/contains.

🔑 Backed by HashMap ❌ No order guarantee ✅ 1 null allowed ⚡ O(1) avg ops ❌ Not thread-safe
Add
O(1)
Remove
O(1)
Contains
O(1)
Iterate
O(n)
Example
Set<String> set = new HashSet<>();
set.add("apple");
set.add("apple");   // ignored — duplicate
set.add(null);      // allowed — 1 null
set.contains("apple");  // true — O(1)
set.remove("apple");    // O(1)
set.size();              // 2 (apple + null)
⚠️ Always override both hashCode() and equals() when using custom objects in HashSet. Breaking the contract causes wrong behavior.
🔗

LinkedHashSet<E>

Insertion-ordered HashSet
class LinkedHashSet<E> ▶

Extends HashSet. Maintains a doubly-linked list through all entries to preserve insertion order. Slightly slower than HashSet due to linked list overhead.

✅ Insertion order preserved ⚡ O(1) avg ops ✅ 1 null allowed ❌ Not thread-safe
Example
Set<String> set = new LinkedHashSet<>();
set.add("banana");
set.add("apple");
set.add("cherry");
// Iterates in insertion order: banana, apple, cherry
set.forEach(System.out::println);
💡 Use LinkedHashSet when you need a Set and want to maintain the order in which elements were inserted — for example, deduplicating a list while preserving order.
🌲

TreeSet<E>

Red-Black Tree — sorted, navigable
class TreeSet<E> ▶

Implements NavigableSet using a Red-Black tree. Elements are always stored in sorted order (natural ordering or custom Comparator). All core ops are O(log n).

🌲 Red-Black tree internally ✅ Always sorted ❌ No null (NullPointerException) ⏱️ O(log n) ops ❌ Not thread-safe
Add
O(log n)
Remove
O(log n)
Contains
O(log n)
First/Last
O(log n)
NavigableSet API — the key differentiator
TreeSet<Integer> ts = new TreeSet<>(List.of(3,1,4,1,5,9,2));
// Stored as: [1, 2, 3, 4, 5, 9]

ts.first();       // 1 — smallest
ts.last();        // 9 — largest
ts.floor(4);     // 4 — largest element ≤ 4
ts.ceiling(6);   // 9 — smallest element ≥ 6
ts.lower(4);     // 3 — largest element < 4 (strict)
ts.higher(5);    // 9 — smallest element > 5 (strict)
ts.headSet(5);   // [1, 2, 3, 4] — elements < 5
ts.tailSet(5);   // [5, 9] — elements ≥ 5
ts.subSet(2,6);  // [2, 3, 4, 5] — [2, 6)
ts.descendingSet(); // [9, 5, 4, 3, 2, 1]

// Custom order
TreeSet<String> byLen = new TreeSet<>(Comparator.comparingInt(String::length));

Queue<E> Interface

FIFO — First-In First-Out
interface Queue<E> ▶

Extends Collection. Designed for holding elements prior to processing. Typically orders elements in FIFO manner. Methods come in pairs: one throws exception, one returns null/false.

OperationThrows ExceptionReturns null / falseUse

PriorityQueue<E>

Min-Heap — elements retrieved by priority
class PriorityQueue<E> ▶

Implements Queue using a binary heap. poll() always returns the smallest element (min-heap by default). Elements must be Comparable or a Comparator must be provided.

🏔️ Binary Min-Heap ❌ No null elements ❌ Not thread-safe ❌ No ordering guarantee on iteration
offer() / add()
O(log n)
poll() — remove min
O(log n)
peek() — view min
O(1)
contains()
O(n)
Min-Heap, Max-Heap, Custom
// Min-Heap (default)
PriorityQueue<Integer> minH = new PriorityQueue<>();
minH.offer(5); minH.offer(1); minH.offer(3);
minH.poll();  // returns 1

// Max-Heap
PriorityQueue<Integer> maxH = new PriorityQueue<>(Collections.reverseOrder());
// or: new PriorityQueue<>((a, b) -> b - a);
maxH.offer(5); maxH.offer(1); maxH.offer(3);
maxH.poll();  // returns 5

// Custom — sort by string length
PriorityQueue<String> pq = new PriorityQueue<>(
    Comparator.comparingInt(String::length)
);

// Top-K pattern (keep K smallest)
PriorityQueue<Integer> topK = new PriorityQueue<>(Collections.reverseOrder());
for (int n : nums) {
    topK.offer(n);
    if (topK.size() > k) topK.poll();
}

ArrayDeque<E>

Resizable-array Deque — faster than LinkedList
class ArrayDeque<E> ▶

Resizable-array implementation of Deque. No capacity restrictions; faster than LinkedList for stack/queue use. Preferred over Stack class.

❌ No null elements ⚡ Faster than LinkedList ❌ Not thread-safe ✅ Preferred Stack impl
As Stack and Queue
Deque<Integer> dq = new ArrayDeque<>();

// As Stack (LIFO) — preferred over Stack class
dq.push(1);    // addFirst
dq.push(2);
dq.peek();    // 2 — view top
dq.pop();     // 2 — remove top

// As Queue (FIFO) — preferred over LinkedList
dq.offer(10); // addLast
dq.offer(20);
dq.poll();    // 10 — remove front

// Sliding window (monotonic deque)
dq.peekFirst();  // max/min of window
dq.pollFirst(); dq.pollLast();

HashMap<K,V>

Hash table — most used Map, must know internals
class HashMap<K,V> ▶

The most-used Map implementation. Hash table with bucket array. O(1) average operations. Allows one null key and multiple null values. No order guarantee.

📦 Default capacity: 16 📊 Load factor: 0.75 ✅ 1 null key allowed ❌ No order ❌ Not thread-safe
🔬 Internal Working (FAANG favorite)
PropertyValue
Default initial capacity16
Default load factor0.75
Resize triggerwhen size > capacity × 0.75
Resize actionDoubles capacity, rehashes all entries
Collision: small bucketLinked list (< 8 entries)
Collision: large bucketRed-Black Tree (≥ 8 entries, Java 8+)
Essential Java 8 Map Methods
Map<String, Integer> map = new HashMap<>();

// Safe get with default
map.getOrDefault("key", 0);

// Put only if absent
map.putIfAbsent("a", 1);

// Compute if key absent — lazy init
map.computeIfAbsent("list", k -> new ArrayList<>()).add("value");

// Merge — word count pattern
map.merge(word, 1, Integer::sum);

// Update existing
map.computeIfPresent("key", (k, v) -> v + 1);

// Iteration — most efficient
for (Map.Entry<String,Integer> e : map.entrySet()) {
    System.out.println(e.getKey() + " → " + e.getValue());
}
map.forEach((k, v) -> System.out.println(k + " → " + v)); // Java 8

LinkedHashMap<K,V>

Insertion-ordered HashMap — LRU Cache foundation
class LinkedHashMap<K,V> ▶

Extends HashMap. Maintains a doubly-linked list through entries to preserve either insertion order or access order. Famous for enabling an LRU Cache in one override.

✅ Insertion order (default) ✅ Access order (optional) ✅ 1 null key allowed ⚡ O(1) avg ops
LRU Cache with LinkedHashMap
// LRU Cache — the classic interview question!
int capacity = 3;
Map<Integer,Integer> lru = new LinkedHashMap<>(capacity, 0.75f, true) {
    // accessOrder=true → least-recently-used at head
    protected boolean removeEldestEntry(Map.Entry<Integer,Integer> eldest) {
        return size() > capacity;  // evict when over capacity
    }
};

lru.put(1, 10);
lru.put(2, 20);
lru.put(3, 30);
lru.get(1);       // accesses 1 → moves to tail (most recent)
lru.put(4, 40);   // evicts 2 (least recently used)
💡 LinkedHashMap(capacity, loadFactor, true) — the third parameter enables access order mode, making it perfect for LRU cache implementation. This is a very common interview question!

TreeMap<K,V>

Red-Black Tree Map — sorted keys, range queries
class TreeMap<K,V> ▶

Implements NavigableMap using a Red-Black tree. Keys are always sorted. O(log n) for all operations. No null keys allowed.

🌲 Red-Black tree ✅ Keys always sorted ❌ No null keys ⏱️ O(log n)
NavigableMap API
TreeMap<Integer,String> tm = new TreeMap<>();
tm.put(1,"one"); tm.put(3,"three"); tm.put(5,"five"); tm.put(7,"seven");

tm.firstKey();       // 1
tm.lastKey();        // 7
tm.floorKey(4);     // 3 — largest key ≤ 4
tm.ceilingKey(4);   // 5 — smallest key ≥ 4
tm.lowerKey(5);     // 3 — largest key < 5
tm.higherKey(5);    // 7 — smallest key > 5
tm.headMap(5);      // {1=one, 3=three}
tm.tailMap(5);      // {5=five, 7=seven}
tm.subMap(1,5);     // {1=one, 3=three}
tm.descendingMap(); // reversed view

Comparable vs Comparator

Natural ordering vs Custom ordering

Comparable<T>

  • Interface in java.lang
  • Defines natural ordering
  • Implemented inside the class
  • Method: compareTo(T o)
  • Only one sort sequence
  • Used by: Collections.sort(), TreeSet/TreeMap

Comparator<T>

  • Interface in java.util
  • Defines custom ordering
  • Implemented externally
  • Method: compare(T o1, T o2)
  • Multiple sort sequences possible
  • @FunctionalInterface — use lambdas!
Full Examples
// ── Comparable — class defines natural order ──
class Student implements Comparable<Student> {
    String name; int age, gpa;

    public int compareTo(Student other) {
        return Integer.compare(this.age, other.age); // natural: by age
    }
}
Collections.sort(students); // uses compareTo

// ── Comparator — external, flexible ──
Comparator<Student> byName = (a, b) -> a.name.compareTo(b.name);
Comparator<Student> byGpaDesc = Comparator.comparingDouble(Student::getGpa).reversed();

// Chaining — sort by age, then name if age ties
Comparator<Student> multi = Comparator
    .comparingInt(Student::getAge)
    .thenComparing(Student::getName);

students.sort(multi);

Fail-Fast vs Fail-Safe

Iterator behavior during concurrent modification
FeatureFail-FastFail-Safe
Works onOriginal collectionClone / snapshot of collection
Exception on modification❌ ConcurrentModificationException✅ No exception
MemoryLessMore (copy overhead)
Reflects live changesYesNo (iterates old snapshot)
ExamplesArrayList, HashMap, HashSet, TreeMapCopyOnWriteArrayList, ConcurrentHashMap
Safe Removal Patterns
List<String> list = new ArrayList<>(List.of("a","b","c","b"));

// ❌ WRONG — ConcurrentModificationException
for (String s : list) {
    if (s.equals("b")) list.remove(s);
}

// ✅ Option 1: Iterator.remove()
Iterator<String> it = list.iterator();
while (it.hasNext()) {
    if (it.next().equals("b")) it.remove();
}

// ✅ Option 2: removeIf — Java 8, cleanest
list.removeIf(s -> s.equals("b"));

// ✅ Option 3: collect to new list (stream)
list = list.stream().filter(s -> !s.equals("b")).collect(Collectors.toList());

Top Interview Q&A

Click each question to reveal the answer
HashMapHow does HashMap work internally?▶
1. Calls key.hashCode(), then spreads bits with an internal hash() function. 2. Computes bucket index: index = hash & (n-1). 3. Places entry in the bucket. 4. Collision? Chains as a LinkedList. 5. Java 8+: if a bucket grows to 8+ entries, converts to a Red-Black Tree (O(log n) vs O(n)). 6. Resizes when size > capacity × 0.75 by doubling capacity and rehashing all entries.
HashSetHow is HashSet implemented internally?▶
HashSet is backed by a HashMap<E, Object>. When you call set.add(e), it calls map.put(e, PRESENT) where PRESENT is a static dummy Object. The uniqueness of elements comes from the uniqueness of HashMap keys.
ContractWhat happens if you don't override hashCode() with equals()?▶
Logically equal objects (same equals()) may have different hashCode() values, landing in different buckets. This means:
• HashSet will store duplicates that are logically equal
• HashMap.get(key) will return null even when the key logically exists
The contract: if a.equals(b) is true, then a.hashCode() == b.hashCode() must also be true. Always override both.
ConcurrencyConcurrentHashMap vs Collections.synchronizedMap()?▶
synchronizedMap wraps the entire map and locks the whole object on every read and write — one thread at a time.
ConcurrentHashMap uses bucket-level locking (Java 8: CAS + synchronized blocks per bucket) — multiple threads can read/write different buckets simultaneously.
→ Prefer ConcurrentHashMap for high-concurrency scenarios. synchronizedMap is only useful when you need to synchronize compound operations on the map itself.
OrderDoes Map extend Collection?▶
No. Map is a completely separate hierarchy. It does not extend Collection or Iterable. This is a classic trick question. The two root interfaces are Collection<E> (for List, Set, Queue) and Map<K,V> (for key-value stores).
IteratorWhat is ConcurrentModificationException and how to avoid it?▶
Thrown by a fail-fast iterator when the underlying collection is structurally modified (add/remove) while iterating — except through the iterator's own remove().
Avoid by: (1) using iterator.remove(), (2) using list.removeIf(predicate) (Java 8), (3) collecting to a new list via stream, or (4) using fail-safe collections like CopyOnWriteArrayList.
Queuepoll() vs remove() in Queue?▶
remove() throws NoSuchElementException if the queue is empty.
poll() returns null if the queue is empty.
In production always prefer poll() and check for null, unless you're certain the queue is non-empty and want the failure to be loud.
SortingComparable vs Comparator — when to use each?▶
Use Comparable when the class has a single obvious natural ordering that belongs to the class itself (e.g., Integer, String). Implement compareTo() inside the class.
Use Comparator when you need multiple orderings, or you're sorting third-party classes you can't modify. Pass it to sort(), TreeSet, or PriorityQueue. Since Java 8, use lambdas: Comparator.comparingInt(Person::getAge).reversed().
LRUHow to implement an LRU Cache in Java?▶
Use LinkedHashMap with accessOrder = true and override removeEldestEntry():
new LinkedHashMap<>(capacity, 0.75f, true) { protected boolean removeEldestEntry(Map.Entry e) { return size() > capacity; } }
With access order enabled, get() moves the accessed entry to the tail, so the head is always the least-recently-used entry, which gets auto-evicted when the map exceeds capacity.
StackWhy prefer ArrayDeque over Stack class?▶
Stack extends Vector which is fully synchronized — unnecessary overhead for single-threaded use. ArrayDeque is resizable, not synchronized, and significantly faster. The Java docs themselves recommend using ArrayDeque as a stack. Use dq.push() / dq.pop() / dq.peek().
